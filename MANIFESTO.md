# PCSA Sovereignty Manifesto

## Purpose

PCSA exists to increase a person's operational sovereignty over their own cognitive workflows.
The system is built to prioritize user agency, local control, and graceful utility under real-world constraints.

## The Universal Floor Mandate

PCSA is designed for users under extreme constraints first:

- Limited hardware
- Low or intermittent connectivity
- No time for manual configuration

A design is not complete if it works only for power users.
A feature is not accepted if it raises the minimum required resources for baseline utility.

## The Authentic Partnership Protocol

PCSA is a partner, not a manipulator.

- It must be transparent about uncertainty and capability limits.
- It must support user decision-making, not replace user agency.
- It must avoid coercive behavior and hidden optimization against user intent.
- It must preserve explicit consent as a core interaction primitive.

## Local-First as a Governance Principle

Local execution is the default operating mode.
External delegation is an exception path that requires policy checks and user trust controls.

Public invariant:

- Baseline utility does not require mandatory cloud dependency.

## Resilience as a Rights Guarantee

Resilience is a user-rights requirement, not an optimization detail.

PCSA must continue to provide useful output during outages and degraded conditions.
Failure modes must degrade utility gracefully instead of terminating interaction.

## Security and Sovereignty

Sovereignty is not only where compute runs. It is also what is disclosed.
PCSA documents its principles and architectural boundaries publicly, while withholding implementation details that could weaken user safety.

See SECURITY_BOUNDARY.md for publication scope and disclosure limits.

## Non-Negotiable Invariants

- Local-first routing for baseline operation
- Explicit consent for external delegation unless user override is explicitly enabled
- Degraded-mode utility under local service instability
- No security-through-obscurity claims; no publication of exploit-enabling internals
- User trust settings must be explicit, reversible, and persistent

## Commitments

- Design for frontline utility over feature spectacle
- Keep operational defaults safe for constrained environments
- Prefer clear guarantees over marketing language
- Evolve policy through transparent public ADRs and security review
